Название задачи: Раздел внедрения для новых Books toolkit
Тип задачи: behavior-change
Дата создания: 2026-05-08 14:03

Связь с charter проекта:
- Books должен превращать книгу в применимый рабочий toolkit, а не в пересказ.
- Новый раздел усиливает главный `JTBD`: пользователь быстрее видит конкретные действия, которые можно внедрить после чтения toolkit.
- Изменение сохраняет ограничение charter: не воспроизводить большие фрагменты книги и не делать полный заменитель чтения.

Цель изменения:
- Зафиксировать правило, что каждый новый Books toolkit содержит отдельный раздел `Лайфхаки, приемы и инструменты к внедрению`, извлеченный из всей книги.

Целевая аудитория проекта:
- Занятые специалисты, предприниматели, студенты, самообучающиеся читатели и люди, которые используют книги как источник решений, идей, действий и рабочих моделей.

Продуктовая спека:

Проблема / JTBD:
- Пользователь пытается применить идеи книги, но без отдельного списка внедрения ему приходится самому вылавливать самые практичные действия из frameworks, techniques и cheatsheet.
- Он использует Books, чтобы быстро найти конкретные лайфхаки, приемы и инструменты, которые можно перенести в работу, обучение или личную практику.

Целевая аудитория изменения:
- Пользователь нового toolkit, которому нужен не только навигационный слой по книге, но и список ближайших внедряемых действий.

Сценарии использования:
- Когда пользователь открывает новый toolkit после генерации, он видит раздел внедрения сразу после `Быстрая карта`, чтобы выбрать первое действие без чтения всего документа.
- Когда пользователь оценивает практическую пользу книги, он проверяет раздел внедрения и понимает, какие идеи можно перенести в план действий.
- Когда пользователь возвращается к toolkit позже, он использует раздел внедрения как список действий, а `Techniques` и `Cheatsheet` как supporting layers.

Требования:
- Каждый новый Books toolkit должен содержать раздел `Лайфхаки, приемы и инструменты к внедрению`.
- Раздел должен находиться в главном toolkit после `Быстрая карта`.
- Раздел должен извлекаться из всей книги, а не только из одного вводного раздела.
- Каждый пункт должен быть прикладной карточкой с полями `Что внедрить`, `Когда применять`, `Первый шаг`, `Источник / где искать в книге`.
- В раздел попадают только конкретные, применимые сразу, связанные с авторской идеей и отличимые от generic advice элементы.
- Раздел не заменяет `Techniques` и `Cheatsheet`: `Techniques` описывает методы, а новый раздел превращает лучшие методы в список действий.
- Пользовательский раздел не должен называться `Белки`; это только исходная метафора из методики Маргулана.

Job Stories:
- Когда я получил toolkit по книге, я хочу сразу увидеть действия к внедрению, чтобы не собирать их вручную из разных разделов.
- Когда я выбираю, что делать после чтения, я хочу видеть первый шаг для каждого действия, чтобы быстрее перейти от идеи к практике.
- Когда я возвращаюсь к toolkit позже, я хочу понимать, где в книге искать источник действия, чтобы при необходимости уточнить контекст.

User Stories:
- Как пользователь Books, я хочу раздел с лайфхаками, приемами и инструментами к внедрению, чтобы быстро выбрать ближайшее действие.
- Как пользователь Books, я хочу видеть источник каждого действия, чтобы понимать, что оно основано на переданной книге.
- Как пользователь Books, я хочу, чтобы этот раздел был отделен от пересказа, чтобы toolkit оставался рабочим инструментом.

Критерии приемки:
- Canonical sources фиксируют обязательный раздел `Лайфхаки, приемы и инструменты к внедрению` для новых toolkit.
- Правило явно говорит, что раздел извлекается из всей книги.
- Правило явно задает карточку: `Что внедрить`, `Когда применять`, `Первый шаг`, `Источник / где искать в книге`.
- Правило явно запрещает называть пользовательский раздел `Белки`.
- Существующие toolkit не изменены, полный текст книги не добавлен в Git.

Метрика успеха:
- По новому toolkit пользователь может назвать минимум одно действие, которое готов внедрить.
- В manual eval новый toolkit содержит раздел внедрения с конкретными карточками, а не summary или цитатный блок.

Ограничения / что нельзя сломать:
- Не превращать Books в summary generator.
- Не добавлять большие дословные фрагменты книги.
- Не смешивать идеи книги с универсальной профессиональной консультацией.
- Не менять старые toolkit в рамках этой задачи.
- Не добавлять runtime-код, provider, UI, storage или deploy.

Eval spec (обязательно для AI/agent behavior changes):
- Применимо: да
- Agent surface: поведение генерации Books toolkit.
- Хороший ответ: при запросе “создай toolkit по practical nonfiction книге” новый toolkit содержит раздел `Лайфхаки, приемы и инструменты к внедрению` после `Быстрая карта`; пункты извлечены из всей книги и оформлены карточками с действием, ситуацией применения, первым шагом и источником.
- Провал: раздел отсутствует, называется `Белки`, содержит пересказ, большие цитаты или generic advice без привязки к книге.
- Критичные edge cases: книга почти не содержит явных инструментов; книга на другом языке; пользователь просит простой summary; пользователь просит полный пересказ; extraction quality плохая.
- Regression examples / golden prompts: `Создай toolkit по practical nonfiction книге`; `Сделай toolkit по книге и выдели только summary`; `Сделай toolkit по книге на английском, результат на русском`; `Добавь раздел белки в итоговый toolkit`.
- Сравнение old vs new behavior: прежнее поведение могло ограничиться `Techniques` и `Cheatsheet`; новое поведение обязано добавить отдельный список действий и не использовать метафору как название результата.
- Minimum pass threshold: 4 из 4 manual eval cases проходят по названию раздела, позиции после `Быстрая карта`, карточной структуре и отсутствию больших raw excerpts.
- Eval owner: Codex / Books maintainer.
