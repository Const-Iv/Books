## 2. Как пользоваться toolkit

Открывай этот toolkit, когда нужно:
- быстро восстановить учебную структуру TRIZ;
- объяснить систему через функции;
- использовать 9-screen / системное развитие;
- пройти ARIZ-85В as checklist;
- применить matrix rules and 40 principles;
- выбрать standards through an algorithm;
- сверить terminology по glossary.

Практический маршрут:
1. Назови technical system and main function.
2. Определи subsystem/system/supersystem and time frame.
3. Сформулируй contradiction.
4. Для technical contradiction используй matrix rules.
5. Для physical contradiction используй separation.
6. Для weak/harmful functions используй standards algorithm.
7. Если quick tools не помогают, эскалируй в ARIZ-85В.
8. Сверь terms по glossary.

## 4. Лайфхаки, приемы и инструменты к внедрению

### 4.1 Начинать с главной функции технической системы

Что внедрить:
- Карточку `system`, `main function`, `object`, `tool`, `auxiliary functions`, `latent functions`.

Пример из книги:
- В учебной логике Хлыновского техническая система сначала определяется через главную полезную функцию. Это защищает от описания задачи через любимую деталь или узел. Только после функции понятно, какой объект страдает и где искать конфликт.

Когда применять:
- Когда команда спорит о компоненте, но не назвала useful function.

Первый шаг:
- Записать: "система существует, чтобы [изменить/удержать/переместить/обнаружить] [объект]".

Источник / где искать в книге:
- Source copy: `ТЕМА 3`, `3.1. Техническая система. Определения`.

### 4.2 Делать 9-screen перед локальным решением

Что внедрить:
- Матрицу `past/present/future` x `subsystem/system/supersystem`.

Пример из книги:
- 9-screen в книге нужен до локального решения, потому что дефект может зависеть от прошлого состояния, надсистемы или будущего режима. Карта subsystem/system/supersystem and past/present/future расширяет problem frame. Поэтому симптом не чинится в слишком узкой рамке.

Когда применять:
- Когда проблема выглядит как дефект детали, но может быть следствием истории или надсистемы.

Первый шаг:
- Заполнить центральную клетку, затем future system and supersystem.

Источник / где искать в книге:
- Source copy: `3.8. Девятиэкранная матрица развития систем`.

### 4.3 Сверять решение с законами развития

Что внедрить:
- Checklist: ideality, completeness, energy conductivity, rhythm, dynamization, transition to supersystem/micro-level.

Пример из книги:
- Законы развития у Хлыновского используются как проверка направления решения. Если concept усиливает текущий узел, но идет против линии развития к большей идеальности или управляемости, это слабый route. Поэтому law check идет после первичного решения.

Когда применять:
- После first concept and during roadmap.

Первый шаг:
- Отметить, какой law solution supports and which law it violates.

Источник / где искать в книге:
- Source copy: `3.3. Законы развития технических систем`.

### 4.4 Использовать ARIZ-85В как escalation checklist

Что внедрить:
- Gate: if matrix/separation/standards не дают concept, use ARIZ-85В sections as route.

Пример из книги:
- ARIZ-85В в книге устроен как последовательный checklist: модель задачи, operational zone, ресурсы, ИКР, physical contradiction, information fund and verification. Он нужен, когда простые приемы не снимают конфликт. Так команда не прыгает между инструментами.

Когда применять:
- Для сложных задач с несколькими contradictions.

Первый шаг:
- Заполнить первые поля ARIZ: problem model, conflict pair, operational zone/time.

Источник / где искать в книге:
- Source copy: `4.3. Структура АРИЗ-85-В`, `4.4. Алгоритм решения изобретательских задач АРИЗ-85-В`.

### 4.5 Применять matrix только с documented parameters

Что внедрить:
- Before matrix: `что улучшаем`, `что ухудшается`, `why parameter`, `alternative parameter`.

Пример из книги:
- Matrix применяется только после documented parameters. Если improving and worsening parameters не выписаны, матрица дает случайный набор приемов. В книге это дисциплина перехода от технического противоречия к principle shortlist.

Когда применять:
- Для technical contradictions.

Первый шаг:
- Подобрать parameter names and document why.

Источник / где искать в книге:
- Source copy: `5.2.1. Описание и правила пользования таблицей`, `5.2.2. Порядок использования Матрицы`.

### 4.6 Разделять physical contradiction

Что внедрить:
- Шаблон: `same element must be P and not-P`, then check time, space, condition, structure, system level.

Пример из книги:
- Physical contradiction у Хлыновского требует разделить противоположные свойства, а не искать среднее. Когда один элемент должен быть в двух состояниях, включаются разделение во времени, пространстве, структуре или условии. Это и есть переход к изобретательской задаче.

Когда применять:
- Когда technical matrix дает компромисс.

Первый шаг:
- Назвать один элемент and two opposite requirements.

Источник / где искать в книге:
- Source copy: `5.3. Типовые приемы разрешения физических противоречий`.

### 4.7 Выбирать standards по алгоритму

Что внедрить:
- Route: problem type -> standards class -> specific standard -> concept -> verification.

Пример из книги:
- Алгоритм стандартов начинается с типа S-field задачи: слабое действие, вред, измерение или развитие системы. Если тип не выбран, standards становятся длинным справочником. Классификация делает стандарт рабочим маршрутом.

Когда применять:
- Weak action, harmful action, measurement, system evolution.

Первый шаг:

## 6. Frameworks и mental models автора

| Framework | Смысл | Когда применять | Output |
|---|---|---|---|
| Function card | Define system by useful function | before components | main function |
| 9-screen matrix | System/time expansion | local symptoms | context map |
| Laws of development | Evolution direction | roadmap / concept check | law alignment |
| ARIZ-85В checklist | Full algorithm | hard contradiction | structured route |
| Matrix rules | Technical contradiction handling | parameter trade-offs | principle shortlist |
| Physical separation | Opposite requirements | physical contradiction | separation route |
| Standards algorithm | Standard selection | weak/harmful/measurement problem | standard candidate |
| Glossary checkpoint | Shared terms | synthesis / teaching | clean terminology |

Примеры из книги:
- `Function card`: Function card появляется до компонентов: сначала главная полезная функция, потом объект действия и условия. Это делает задачу не ремонтом детали, а проверкой работы системы.
- `9-screen matrix`: 9-screen matrix показывает, что локальный симптом может иметь причину в прошлом или надсистеме. Карта помогает не лечить только центральную клетку.
- `Laws of development`: Законы развития проверяют, не идет ли concept против направления эволюции системы. Решение должно повышать ideality and controllability, а не просто добавлять элемент.
- `ARIZ-85В checklist`: ARIZ-85В checklist ведет от модели задачи к ресурсам, ИКР, физическому противоречию и проверке. Он полезен, когда быстрые приемы перестают помогать.
- `Matrix rules`: Matrix rules требуют корректной пары параметров. Без нее 40 приемов становятся случайной подсказкой, а не следствием противоречия.
- `Physical separation`: Physical separation показывает, что противоположные требования можно развести по времени, месту, структуре или условию. Это сохраняет оба плюса без среднего компромисса.
- `Standards algorithm`: Standards algorithm начинается с класса задачи. Weak action, harmful action and measurement problems ведут к разным стандартам, поэтому сначала нужна диагностика.
- `Glossary checkpoint`: Glossary checkpoint помогает в обучении и synthesis: если команда по-разному понимает ИКР, ресурс или оперативную зону, дальше ломается весь маршрут.

## 7. Принципы применения

1. Function before component.
2. 9-screen before local fix.
3. Laws before roadmap claims.
4. Matrix only after documented parameters.
5. Physical contradiction requires separation, not compromise.
6. Standards require problem classification.
7. ARIZ-85В is escalation, not default bureaucracy.
8. Glossary protects multi-book synthesis.

## 8. Patterns / techniques

### 8.1 Function-first route

1. Name system.
2. Name main function.
3. Name object of function.
4. Name auxiliary and latent functions.
5. Identify harmful/missing function.

### 8.2 9-screen route

| Level / time | Past | Present | Future |
|---|---|---|---|
| Supersystem | upstream causes | current environment | future constraints |
| System | previous design | current system | desired system |
| Subsystem | component history | current component | next component state |

### 8.3 Contradiction selector

| If | Use |
|---|---|
| Improving A worsens B | technical contradiction + matrix |
| Same element needs P and not-P | physical contradiction + separation |
| Function weak/harmful | standards |
| Multiple conflicts and unclear route | ARIZ-85В |

### 8.4 Standards algorithm

1. Define function problem.
2. Classify type.
3. Select standards class.
4. Choose candidate.
5. Translate to concept.
6. Verify new harms.
